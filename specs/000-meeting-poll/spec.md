## 1. 배경 / 문제 정의
Meeet은 여러 사람이 참여하는 미팅 하나의 시간을 빠르게 정하기 위한 일회성 도구다.
회원가입 없이 누구나 바로 후보 시간 여러 개로 미팅 폴을 만들 수 있고, 생성된 폴마다
고유한 공개 링크가 발급된다. 참가자는 그 링크로 들어와 이름을 입력하고 자신이
가능한 시간을 표시(투표)한다. 모두의 응답을 한눈에 모아 보여줘서, 가장 많은 사람이
가능한 시간을 빠르게 찾고 확정할 수 있게 한다.

이 도구에는 사이트 차원의 계정/로그인 개념이 없다. 폴을 만든 사람도, 폴에 참여하는
사람도 별도 회원가입을 하지 않는다 — "참여하려면 이름을 입력한다"가 유일한 등록 절차다.
폴의 공개 링크(토큰)를 아는 것 자체가 그 폴에 대한 접근/참여/확정 권한이다.
(Calendly처럼 폴 생성자가 반복되는 가용 시간을 영구 등록해두는 모델이 아니다 — 그 모델은 비목표.)

## 2. 목표 (Goals)
- 누구나 로그인 없이 제목과 후보 시간 슬롯 여러 개로 미팅 폴을 즉시 만들 수 있다.
- 폴 생성 시 추측 불가능한 토큰 기반의 공개 링크가 발급되어 바로 공유할 수 있다.
- 참가자는 그 링크로 들어와 이름을 입력하고 가능한 후보 시간을 표시할 수 있다.
- 누구나(폴 생성자, 참가자, 링크를 받은 누구든) 폴 페이지에서 후보 시간별 가능 인원 수와
  명단을 한눈에 볼 수 있다.
- 링크를 아는 사람은 누구나 후보 시간 중 하나를 최종 확정하여 폴을 마감할 수 있다.

## 3. 비목표 (Non-Goals)
- 사이트 차원의 회원가입/로그인, 사용자 계정, 비밀번호 — 폴 생성자와 참가자 모두 계정이 없다.
  (이하 "호스트"라는 표현은 쓰지 않고 "폴 생성자"로 통일한다.)
- 폴 생성자 전용 대시보드("내가 만든 폴 목록") — 계정이 없으므로 존재할 수 없다.
- 반복되는 주간 가용 시간 등록 (Calendly식 모델).
- 참가자 이메일/푸시 알림.
- 외부 캘린더 연동(Google Calendar 등).
- "가능 / 불가능 / 어쩌면" 같은 3단계 응답 — MVP는 가능/불가능 2단계만.
- 폴 자동 만료(시간 경과 시 자동 마감) — MVP는 링크 소지자의 수동 확정만 지원.
- 기간 입력 후 후보 시간 자동 분할 (예: "다음 주 평일 9~18시" → 슬롯 자동 생성) — 하나씩 직접 입력하는 방식만 지원.
- 참가자 응답 수정 시 익명 편집 토큰을 통한 본인 인증 — 이름 일치만으로 덮어쓰기를 허용한다 (보안보다 단순함 우선).
- 폴 생성 후 후보 시간 추가/삭제, 폴 자체의 삭제 — MVP는 생성만 지원하고 수정/삭제는 지원하지 않는다.

## 4. 사용자 시나리오 (User Stories)
- P1: As a 폴 생성자, I want to 가입 없이 바로 미팅 제목과 후보 시간 여러 개를 등록하고, so that 곧바로 참가자들에게 투표를 요청할 수 있다.
- P1: As a 폴 생성자, I want to 생성 직후 공개 링크를 받고, so that 참가자에게 바로 공유할 수 있다.
- P1: As a 참가자, I want to 링크에 접속해 이름만 입력하고 가능한 시간을 체크하고, so that 별도 가입 없이 빠르게 참여할 수 있다.
- P1: As a 링크를 가진 사람, I want to 각 후보 시간별 가능 인원 수를 한눈에 보고, so that 가장 많은 사람이 가능한 시간을 쉽게 찾을 수 있다.
- P1: As a 링크를 가진 사람, I want to 최종 시간을 확정하고, so that 폴이 끝나고 모두에게 결과가 명확해진다.
- P2: As a 참가자, I want to 이미 제출한 내 응답을 나중에 다시 와서 수정하고, so that 일정이 바뀌었을 때 반영할 수 있다.

## 5. 기능 요구사항 (Functional Requirements)
- FR-1: 누구나 로그인/회원가입 없이 미팅 폴을 생성할 수 있다.
- FR-2: 미팅 폴은 제목(필수), 설명(선택), 후보 시간 슬롯 목록(1개 이상)을 가진다.
- FR-3: 각 후보 시간 슬롯은 시작 시각과 종료 시각을 가지며(종료 시각 > 시작 시각), 시작 시각이 과거인 슬롯은 생성할 수 없다.
- FR-4: 폴 생성 시 추측 불가능한 공개 토큰이 발급된다. 폴 조회, 참가자 응답, 확정을 포함한
  모든 동작은 이 토큰을 아는 것만으로 가능하다 — 별도의 생성자 권한 구분은 없다.
- FR-5: 공개 폴 페이지는 폴 제목, 설명, 후보 시간 목록, 현재까지의 참여 현황(슬롯별 가능 인원/명단)을 누구나 볼 수 있다.
- FR-6: 참가자는 이름을 입력하고, 각 후보 시간에 대해 가능/불가능을 표시한 뒤 응답을 제출할 수 있다.
- FR-7: 참가자가 같은 폴에 동일한 이름으로 다시 응답을 제출하면 기존 응답을 덮어쓴다. 이름만으로 식별하며 별도 인증 토큰은 두지 않는다 (트레이드오프는 §8 참고).
- FR-8: 폴 페이지는 후보 시간별 가능 인원 수를 집계해 내림차순 또는 시간순으로 보여준다.
- FR-9: 공개 토큰을 아는 사람은 누구나 후보 시간 중 하나를 확정 시간으로 선택해 폴 상태를 "확정(confirmed)"으로 변경할 수 있다.
- FR-10: 확정된 폴은 더 이상 참가자 응답이나 재확정을 받지 않는다.
- FR-11: 모든 사용자 입력(폴 생성, 참가자 응답, 확정)은 Zod 스키마로 검증한다.
- FR-12: 폴 생성 시 후보 시간 슬롯은 하나씩 직접 입력한다. 기간을 주고 자동으로 슬롯을 분할해주는 기능은 MVP 범위 밖이다.
- FR-13: 후보 시간과 참여 현황은 서버에 UTC로 저장하고, 클라이언트(공개 폴 페이지)에서는 각 사용자의 브라우저 타임존으로 변환해 표시한다.

## 6. 데이터 모델 (Data Model)

| 엔티티 | 필드 | 타입 | 제약조건 |
|---|---|---|---|
| MeetingPoll | id | uuid | PK |
| | title | text | not null |
| | description | text | nullable |
| | publicToken | text | unique, not null (랜덤, 추측 불가능 — 조회/응답/확정 모든 권한의 근거) |
| | status | enum(open, confirmed) | not null, default open |
| | confirmedSlotId | uuid | FK -> PollSlot.id, nullable |
| | createdAt | timestamp | not null, default now |
| PollSlot | id | uuid | PK |
| | pollId | uuid | FK -> MeetingPoll.id, not null |
| | startsAt | timestamp (tz, UTC 저장) | not null |
| | endsAt | timestamp (tz, UTC 저장) | not null, > startsAt |
| Participant | id | uuid | PK |
| | pollId | uuid | FK -> MeetingPoll.id, not null |
| | name | text | not null |
| | createdAt | timestamp | not null, default now |
| | (unique) | pollId + name | 같은 폴 내에서 이름 중복 방지 — FR-7의 "이름으로 덮어쓰기"가 성립하려면 필수 |
| ParticipantAvailability | participantId | uuid | FK -> Participant.id, not null (PK 일부) |
| | pollSlotId | uuid | FK -> PollSlot.id, not null (PK 일부) |
| | (PK) | participantId + pollSlotId | 복합 PK — "가능"으로 표시한 슬롯만 행으로 존재 |

User 테이블은 없다 — 이 프로젝트에는 인증 대상이 되는 계정 엔티티가 존재하지 않는다.

## 7. API / 인터페이스 개요

페이지 (App Router):
- `GET /` — 폴 생성 페이지. 제목/설명/후보 시간 슬롯을 입력하는 폼이 여기 있다 (서비스의 진입점).
- 폴 생성 성공 후에는 발급된 공개 링크(`/p/{token}`)를 복사할 수 있는 공유 화면을 보여준다.
  (별도 라우트로 둘지, 생성 페이지 내 상태 전환으로 둘지는 `plan.md`에서 결정.)
- `GET /p/{token}` — 공개 폴 페이지. 폴 내용 조회, 참가자 응답 입력, 확정을 모두 이 페이지에서 수행한다 (인증 불필요).

API:
- `POST /api/polls` — 미팅 폴 생성. 성공 시 발급된 `publicToken`을 반환한다 (인증 불필요, 누구나 호출 가능).
- `POST /api/polls/{token}/responses` — 참가자 응답 제출/수정 (인증 불필요)
- `POST /api/polls/{token}/confirm` — 후보 시간 확정 (인증 불필요, 토큰 소지자면 누구나)

## 8. 엣지 케이스 & 에러 처리
- 후보 시간이 1개도 없는 폴 생성 요청은 거부한다.
- 과거 시각으로 후보 시간을 등록하려는 요청은 Zod 단계에서 거부한다.
- 확정된(confirmed) 폴에 참가자 응답 제출을 시도하면 409 Conflict를 반환한다.
- 존재하지 않는 publicToken으로 접근하면 404를 반환한다.
- 확정 요청 시 지정한 슬롯이 해당 폴에 속하지 않으면 400을 반환한다.
- 이미 확정된 폴에 재확정을 시도하면 409를 반환한다.
- 동시에 두 사람이 서로 다른 슬롯으로 확정을 시도하면, 먼저 폴 상태를 confirmed로 바꾼 쪽만 성공하고 나중 요청은 409를 반환한다 (status 전이를 원자적으로 처리해 보장한다 — 예: `WHERE status = 'open'` 조건부 갱신).
- 같은 이름의 참가자가 응답을 다시 제출하면 기존 응답을 덮어쓴다 — 타인이 같은 이름으로 응답을 덮어쓸 수 있는 보안 트레이드오프가 있음을 인지하고 수용한다 (FR-7).
- 토큰을 아는 사람은 누구나(원래 만든 사람이 아니어도) 확정까지 할 수 있다는 트레이드오프를 인지하고 수용한다 — 계정이 없으므로 권한을 더 세분화할 방법이 없다 (FR-9).
- 클라이언트의 타임존 판별이 불가능한 경우(JS 비활성 등은 Next.js 앱에서 사실상 발생하지 않음) UTC를 그대로 표시한다.

## 9. 열린 질문 (Open Questions)
- [ ] publicToken의 길이/엔트로피 — 단순 조회 권한이 아니라 확정까지 가능한 권한이므로, 추측 공격에 안전한 충분한 길이(예: 256bit 랜덤)가 필요하다. 구체적인 생성 방식은 `plan.md`에서 결정.
