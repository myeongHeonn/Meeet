# Spec: 미팅 폴 흐름 (Meeting Poll Flow)

- Status: Approved
- Owner: TBD
- Created: 2026-06-30
- Approved: 2026-06-30 (When2meet식 캘린더+격자 모델로 전환 후 재승인)

## 1. 배경 / 문제 정의
Meeet은 여러 사람이 참여하는 미팅 하나의 시간을 빠르게 정하기 위한 일회성 도구다(When2meet식).
회원가입 없이 누구나 바로 미팅 폴을 만들 수 있고, 생성된 폴마다 고유한 공개 링크가 발급된다.

폴 생성자는 **캘린더에서 후보 날짜 여러 개를 고르고, 하루 시간 범위(예: 오전 9시~오후 5시)를
지정**한다. 서버는 이 [날짜 × 시간 범위]를 **30분 격자**로 펼쳐 후보 시간 칸들을 만든다.
참가자는 링크로 들어와 이름을 입력하고, **격자에서 자신이 가능한 시간 칸을 클릭·드래그로 칠한다**.
모두의 응답은 칸마다 가능 인원이 진해지는 **히트맵**으로 한눈에 모여, 가장 많은 사람이 가능한
시간을 빠르게 찾고 확정할 수 있다.

이 도구에는 사이트 차원의 계정/로그인 개념이 없다. 폴을 만든 사람도, 폴에 참여하는
사람도 별도 회원가입을 하지 않는다 — "참여하려면 이름을 입력한다"가 유일한 등록 절차다.
폴의 공개 링크(토큰)를 아는 것 자체가 그 폴에 대한 접근/참여/확정 권한이다.
(Calendly처럼 폴 생성자가 반복되는 가용 시간을 영구 등록해두는 모델이 아니다 — 그 모델은 비목표.)

## 2. 목표 (Goals)
- 누구나 로그인 없이, 캘린더에서 후보 날짜 여러 개 + 하루 시간 범위를 골라 미팅 폴을 즉시 만들 수 있다.
- 서버는 [선택 날짜 × 시간 범위]를 30분 격자 칸으로 펼쳐 저장한다.
- 폴 생성 시 추측 불가능한 토큰 기반의 공개 링크가 발급되어 바로 공유할 수 있다.
- 참가자는 링크로 들어와 이름을 입력하고, 격자에서 가능한 시간 칸을 클릭·드래그로 칠해 응답한다.
- 누구나 폴 페이지에서 칸별 가능 인원을 히트맵으로 보고, 특정 칸에 가능한 사람 명단을 확인할 수 있다.
- 링크를 아는 사람은 누구나 한 시간 칸을 최종 확정하여 폴을 마감할 수 있다.

## 3. 비목표 (Non-Goals)
- 사이트 차원의 회원가입/로그인, 사용자 계정, 비밀번호 — 폴 생성자와 참가자 모두 계정이 없다.
  (이하 "호스트"라는 표현은 쓰지 않고 "폴 생성자"로 통일한다.)
- 폴 생성자 전용 대시보드("내가 만든 폴 목록") — 계정이 없으므로 존재할 수 없다.
- 요일 반복 폴("매주 월/수/금") — MVP는 특정 날짜만 지원한다.
- 날짜만 고르는 올데이(시간 없는) 폴 — MVP는 하루 시간 범위 + 30분 격자를 항상 사용한다.
- 30분 외의 가변 격자 간격(15분/1시간 선택) — MVP는 30분 고정.
- 반복되는 주간 가용 시간 등록 (Calendly식 모델).
- 참가자 이메일/푸시 알림.
- 외부 캘린더 연동(Google Calendar 등).
- "가능 / 불가능 / 어쩌면" 같은 3단계 응답 — MVP는 가능/불가능 2단계만.
- 폴 자동 만료(시간 경과 시 자동 마감) — MVP는 링크 소지자의 수동 확정만 지원.
- 참가자 응답 수정 시 익명 편집 토큰을 통한 본인 인증 — 이름 일치만으로 덮어쓰기를 허용한다 (보안보다 단순함 우선).
- 폴 생성 후 날짜/시간 범위 변경, 폴 자체의 삭제 — MVP는 생성만 지원하고 수정/삭제는 지원하지 않는다.

## 4. 사용자 시나리오 (User Stories)
- P1: As a 폴 생성자, I want to 가입 없이 캘린더에서 후보 날짜들과 하루 시간 범위를 골라 폴을 만들고, so that 곧바로 참가자들에게 투표를 요청할 수 있다.
- P1: As a 폴 생성자, I want to 생성 직후 공개 링크를 받고, so that 참가자에게 바로 공유할 수 있다.
- P1: As a 참가자, I want to 링크에 접속해 이름만 입력하고 격자에서 가능한 시간을 드래그로 칠하고, so that 별도 가입 없이 빠르게 참여할 수 있다.
- P1: As a 링크를 가진 사람, I want to 칸별 가능 인원을 히트맵으로 보고 특정 칸의 가능자 명단을 확인하고, so that 가장 많은 사람이 가능한 시간을 쉽게 찾을 수 있다.
- P1: As a 링크를 가진 사람, I want to 최종 시간을 확정하고, so that 폴이 끝나고 모두에게 결과가 명확해진다.
- P2: As a 참가자, I want to 이미 제출한 내 응답을 나중에 다시 와서 수정하고, so that 일정이 바뀌었을 때 반영할 수 있다.

## 5. 기능 요구사항 (Functional Requirements)
- FR-1: 누구나 로그인/회원가입 없이 미팅 폴을 생성할 수 있다.
- FR-2: 폴 생성 입력은 제목(필수), 설명(선택), 후보 날짜 목록(1개 이상), 하루 시간 범위(시작 시각·종료 시각, 종료 > 시작)로 구성된다.
- FR-3: 서버는 [후보 날짜 × 시간 범위]를 30분 격자로 펼쳐 후보 시간 칸(슬롯)들을 생성·저장한다. 각 칸은 30분 길이다. 과거 날짜는 후보로 넣을 수 없다(과거 날짜 거부). 단 오늘 날짜를 고른 경우 이미 지난 시간대 칸이 일부 남을 수 있는데, MVP에서는 이를 별도로 제거하지 않고 수용한다(폴에서 지난 칸에 가능 표시는 무의미할 뿐 해롭지 않음).
- FR-4: 폴 생성 시 추측 불가능한 공개 토큰이 발급된다. 폴 조회, 참가자 응답, 확정을 포함한 모든 동작은 이 토큰을 아는 것만으로 가능하다 — 별도의 생성자 권한 구분은 없다.
- FR-5: 공개 폴 페이지는 폴 제목, 설명, 날짜×시간 격자, 현재까지의 참여 현황(칸별 가능 인원 히트맵)을 누구나 볼 수 있다.
- FR-6: 참가자는 이름을 입력하고, 격자에서 가능한 칸을 클릭·드래그로 선택(토글)한 뒤 응답을 제출할 수 있다. 칠한 칸 = "가능", 칠하지 않은 칸 = "불가능"(2단계).
- FR-7: 참가자가 같은 폴에 동일한 이름으로 다시 응답을 제출하면 기존 응답을 덮어쓴다. 이름만으로 식별하며 별도 인증 토큰은 두지 않는다 (트레이드오프는 §8 참고).
- FR-8: 폴 페이지는 칸별 가능 인원을 집계해 히트맵(가능 인원이 많을수록 진한 색)으로 보여주고, 특정 칸을 가리키면 그 칸에 가능한 참가자 명단을 보여준다.
- FR-9: 공개 토큰을 아는 사람은 누구나 한 시간 칸을 확정 칸으로 선택해 폴 상태를 "확정(confirmed)"으로 변경할 수 있다.
- FR-10: 확정된 폴은 더 이상 참가자 응답이나 재확정을 받지 않는다.
- FR-11: 모든 사용자 입력(폴 생성, 참가자 응답, 확정)은 Zod 스키마로 검증한다.
- FR-12: 후보 시간 칸·참여 현황은 서버에 UTC로 저장하고, 클라이언트(공개 폴 페이지)에서는 각 사용자의 브라우저 타임존으로 변환해 표시한다. 격자의 날짜 행/시간 열은 뷰어의 브라우저 타임존 기준으로 재구성한다 — 생성 시점의 칸 경계(생성자 타임존 기준 30분)는 UTC로 고정되지만, 표시 시 행/열 배치는 뷰어마다 달라질 수 있다(§6 데이터 모델 노트, §8 타임존 참고).

## 6. 데이터 모델 (Data Model)

| 엔티티 | 필드 | 타입 | 제약조건 |
|---|---|---|---|
| MeetingPoll | id | uuid | PK |
| | title | text | not null |
| | description | text | nullable |
| | publicToken | text | unique, not null (랜덤, 추측 불가능 — 조회/응답/확정 모든 권한의 근거) |
| | status | enum(open, confirmed) | not null, default open |
| | confirmedSlotId | uuid | FK -> PollSlot.id, nullable |
| | createdAt | timestamp | not null, default now |
| PollSlot | id | uuid | PK |
| | pollId | uuid | FK -> MeetingPoll.id, not null |
| | startsAt | timestamp (tz, UTC 저장) | not null (30분 격자 칸의 시작) |
| | endsAt | timestamp (tz, UTC 저장) | not null, = startsAt + 30분 |
| | (unique) | pollId + startsAt | 같은 폴 내 칸 중복 방지 |
| Participant | id | uuid | PK |
| | pollId | uuid | FK -> MeetingPoll.id, not null |
| | name | text | not null |
| | createdAt | timestamp | not null, default now |
| | (unique) | pollId + name | 같은 폴 내에서 이름 중복 방지 — FR-7의 "이름으로 덮어쓰기"가 성립하려면 필수 |
| ParticipantAvailability | participantId | uuid | FK -> Participant.id, not null (PK 일부) |
| | pollSlotId | uuid | FK -> PollSlot.id, not null (PK 일부) |
| | (PK) | participantId + pollSlotId | 복합 PK — "가능"으로 칠한 칸만 행으로 존재 |

- 격자는 `PollSlot` 칸들의 집합으로 표현된다. 별도의 "날짜 목록"이나 "시간 범위" 컬럼은
  두지 않고, 슬롯들에서 파생한다. 단 날짜 행/시간 열로의 분해는 **각 뷰어의 브라우저 타임존
  기준**으로 한다(FR-12) — 같은 폴이라도 뷰어 타임존이 다르면 행/열 배치가 달라질 수 있다
  (자정 부근 칸이 다른 날짜로 넘어가는 등). 격자 간격(30분)은 코드 상수다.
- User 테이블은 없다 — 이 프로젝트에는 인증 대상이 되는 계정 엔티티가 존재하지 않는다.

## 7. API / 인터페이스 개요

페이지 (App Router):
- `GET /` — 폴 생성 페이지. 캘린더로 후보 날짜 선택 + 하루 시간 범위 지정 폼이 여기 있다 (서비스의 진입점).
- 폴 생성 성공 후에는 발급된 공개 링크(`/p/{token}`)를 복사할 수 있는 공유 화면을 보여준다 (생성 페이지 내 상태 전환).
- `GET /p/{token}` — 공개 폴 페이지. 격자 조회, 참가자 응답(드래그 칠하기), 히트맵, 확정을 모두 이 페이지에서 수행한다 (인증 불필요).

API:
- `POST /api/polls` — 미팅 폴 생성. 입력 `{ title, description?, dates[], startTime, endTime, timeZone }`를
  받아 30분 격자 슬롯으로 펼쳐 저장하고, 발급된 `publicToken`을 반환한다 (인증 불필요).
  `timeZone`(IANA 문자열)은 `dates`/`startTime`/`endTime`을 UTC로 변환하는 기준이므로 필수다.
- `POST /api/polls/{token}/responses` — 참가자 응답 제출/수정. `{ name, availableSlotIds[] }` (인증 불필요).
- `POST /api/polls/{token}/confirm` — 후보 칸 확정. `{ slotId }` (인증 불필요, 토큰 소지자면 누구나).

## 8. 엣지 케이스 & 에러 처리
- 후보 날짜가 0개이거나 시간 범위가 비정상(종료 ≤ 시작)인 생성 요청은 Zod 단계에서 거부한다.
- 시간 범위가 30분 단위로 나누어떨어지지 않으면(예: 9:00~9:20) 거부하거나 30분 경계로 내림 처리한다 — 거부로 단순화한다.
- 격자가 지나치게 커지는 입력(날짜를 너무 많이 고르거나 시간 범위가 과도하게 넓음)은 칸 수 상한(MVP: 날짜 ≤ 31, 총 칸 ≤ 1000)으로 거부한다(plan §7 격자 폭증 방어).
- 과거 날짜를 후보로 넣는 생성 요청은 거부한다.
- 확정된(confirmed) 폴에 참가자 응답 제출을 시도하면 409 Conflict를 반환한다.
- 존재하지 않는 publicToken으로 접근하면 404를 반환한다.
- 확정 요청 시 지정한 칸(slotId)이 해당 폴에 속하지 않으면 400을 반환한다.
- 이미 확정된 폴에 재확정을 시도하면 409를 반환한다.
- 동시에 두 사람이 서로 다른 칸으로 확정을 시도하면, 먼저 폴 상태를 confirmed로 바꾼 쪽만 성공하고 나중 요청은 409를 반환한다 (status 전이를 원자적으로 처리해 보장한다 — 예: `WHERE status = 'open'` 조건부 갱신).
- 응답 제출 시 `availableSlotIds`에 이 폴 소속이 아닌 칸이 섞이면 400을 반환한다.
- 같은 이름의 참가자가 응답을 다시 제출하면 기존 응답을 덮어쓴다 — 타인이 같은 이름으로 응답을 덮어쓸 수 있는 보안 트레이드오프가 있음을 인지하고 수용한다 (FR-7).
- 토큰을 아는 사람은 누구나(원래 만든 사람이 아니어도) 확정까지 할 수 있다는 트레이드오프를 인지하고 수용한다 — 계정이 없으므로 권한을 더 세분화할 방법이 없다 (FR-9).
- **타임존**: 슬롯은 UTC로 저장한다. 격자는 폴 생성자의 타임존 기준 30분 경계로 만들어지고, 다른 타임존의 참가자에게는 변환된 로컬 시각으로 라벨을 표시한다(FR-12). 30분 단위가 아닌 타임존(예: UTC+5:30)에서는 라벨이 :00/:30이 아닌 경계로 보일 수 있으나 칸 자체는 동일하게 유지된다(수용).

## 9. 열린 질문 (Open Questions)
- [x] publicToken의 길이/엔트로피 → `plan.md` §7에서 결정: `node:crypto`의 `randomBytes(32).toString("base64url")`(256bit)로 생성한다.
- [x] 폴 생성자의 타임존 → 결정: 생성 시 브라우저 타임존(IANA 문자열)을 `timeZone` 필드로 **필수 전송**한다. 서버가 `dates`/`startTime`/`endTime`을 UTC로 변환하는 데 반드시 필요하므로(이 값 없이는 슬롯을 만들 수 없음) 선택이 아니다. 저장은 UTC 슬롯으로 충분하며 `timeZone` 자체를 폴에 영구 저장하진 않는다.
